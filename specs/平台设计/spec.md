# 平台设计规格说明（Spec）

## 1. 文档目标
- 为 `background / client / install / webset` 提供统一的产品与技术设计基线。
- 以“后台可配置优先、客户端轻更新”为核心，降低后续维护与发版成本。
- 面向怀旧游戏场景，支持“产品展示 + 下载安装 + 环境修复 + 用户互动 + 支付闭环”。

## 2. 设计原则
- 配置驱动：非框架级改动尽量通过后台配置完成，不强依赖客户端升级。
- 模块解耦：账号、支付、内容、版本、下载、修复、互动各模块职责清晰。
- 统一体验：桌面端与门户站的业务能力一致，展示层可按终端差异化。
- 可运维性：后台具备可观测、可回滚、可审计能力。
- 安全合规：最小权限、敏感数据脱敏、下载资源完整性校验。

## 3. 全站业务架构

### 3.1 业务域
- 用户域：注册登录、等级体系、会员状态、资产与订单历史。
- 产品域：游戏本体、MOD、插件、版本包、资源元数据、展示素材。
- 交付域：下载地址编排、渠道管理、安装结果、更新策略。
- 修复域：通用环境修复（VC++/DX）与单游戏专项修复（兼容项、组件开关等）。
- 运营域：首页与栏目配置、轮播配置、公告、留言、推荐位。
- 商业域：支付渠道、订单、对账、退款、风控与回调处理。

### 3.2 系统分层
- 展示层：`client`（Electron + Vue）、`webset`（门户站）。
- 接入层：统一 API 网关与鉴权中间层（可先由 `background` API 模块承载）。
- 业务层：账户、产品、订单、下载、修复、CMS 配置、版本控制服务。
- 数据层：关系型数据库 + 对象存储 + 缓存 + 日志存储。

## 4. 子系统定位与职责

### 4.1 background（后台管理与服务端）
- 基于现有框架代码扩展，不推翻重建。
- 核心职责：
  - 账户管理：用户等级、会员状态、购买资产、设备/安装记录。
  - 产品管理：游戏树结构、版本包、下载地址、素材与文案。
  - 展示配置：门户与客户端页面组件配置、轮播模板与栏目编排。
  - 版本与热更新：客户端版本策略、灰度范围、强制更新开关、回滚点。
  - 支付配置：渠道参数、商品映射、回调验签、异常重试。
  - 运营工具：留言审核、Banner 管理、发布审批、操作审计。

### 4.2 client（Windows 桌面端）
- 核心能力：
  - 账号登录与用户中心（会员信息、购买记录、下载权益）。
  - 游戏库展示与检索、混合轮播（图/文/视频）详情页。
  - 已安装游戏启动器（启动参数、状态管理、快捷操作）。
  - 下载与安装管线（断点续传、校验、安装结果回写）。
  - 修复中心（通用修复包 + 游戏专项修复脚本执行）。
  - 留言互动（发布、回复、举报、审核状态同步）。
- 更新策略：
  - UI 配置、文案、展示结构通过远端配置热更新。
  - 客户端壳层仅在引擎级能力变更时发布新版本。

### 4.3 install（轻量安装器）
- 目标：体积小、依赖少、仅负责下载与安装引导。
- 设计边界：
  - 不承载业务页面与社区功能。
  - 支持安装目录选择、磁盘检测、必要组件检查与拉起客户端。
  - 支持下载镜像源切换、失败重试与基础日志导出。

### 4.4 webset（门户网站）
- 与 client 保持同业务模型，提供更详细的信息浏览与外链传播能力。
- 核心能力：账号体系、产品展示、下载入口、修复说明、留言互动、支付购买。
- 配置方式：页面区块、专题页、推荐位、活动页均由后台统一配置。

## 5. 关键功能设计

### 5.1 用户账户与会员体系
- 账户维度：基础信息、等级、会员有效期、权益包、设备绑定信息。
- 资产维度：已购产品、来源渠道、订单状态、下载次数、安装设备、修复记录。
- 风控维度：异常登录、异常下载、回调异常订单与人工复核入口。

### 5.2 产品展示与内容编排
- 游戏详情结构：简介、配置需求、图文视频轮播、版本说明、常见问题、留言区。
- 资源分类：本体、MOD、插件、补丁、存档包、工具包。
- 展示组件标准化：Banner、宫格、卡片流、时间线、标签过滤器。

### 5.3 下载、安装、修复闭环
- 下载：多渠道地址、优先级路由、节点健康检查、校验值校验。
- 安装：安装向导、依赖检查、进度回写、失败码归因。
- 修复：
  - 通用修复模板：VC++、DirectX、运行库补齐。
  - 专项修复模板：兼容模式、DirectPlay、硬件开关、注册表策略（白名单动作）。
- 全链路状态：下载→安装→首次启动→修复结果统一回传后台。

### 5.4 版本控制与热更新
- 配置版本：页面结构、文案、资源位、推荐规则独立版本号。
- 客户端版本：最小支持版本、灰度组、强制升级阈值。
- 回滚机制：配置与更新包均支持一键回滚及影响范围提示。

### 5.5 支付渠道与订单
- 支付能力：多渠道配置、商品映射、回调验签、状态对账。
- 运营能力：活动价、限时折扣、组合包、会员捆绑商品。
- 稳定性：回调幂等、超时补偿、对账任务、异常工单。

## 6. 数据与配置模型（逻辑级）

### 6.1 核心实体
- 用户：User、UserLevel、Membership、UserDevice。
- 商品：GameProduct、Package、ResourceFile、ProductMedia。
- 交易：Order、PaymentTransaction、Refund、ChannelConfig。
- 交付：DownloadTask、InstallRecord、RepairRecord、UpdatePolicy。
- 内容：PageConfig、ModuleConfig、BannerConfig、Comment、ModerationLog。

### 6.2 配置中心规范
- 配置分层：全局配置 / 终端配置 / 产品配置 / 活动配置。
- 配置发布：草稿、预览、审批、发布、回滚。
- 配置治理：变更审计、版本对比、字段级校验、环境隔离。

## 7. 接口与协议规范
- 统一鉴权：JWT + 刷新令牌机制（后台管理与用户端分域权限）。
- API 规范：统一响应结构、错误码字典、分页协议、幂等键。
- 下载协议：支持断点续传、文件分片校验、签名 URL 过期控制。
- 客户端上报：安装状态、启动状态、修复结果、错误码与机器摘要信息。

## 8. 界面与交互规范
- 风格方向：怀旧科技感 + 现代扁平交互，强调可读性与状态反馈。
- 后台体验：高频任务路径最短化（2~3 步完成配置发布）。
- 桌面端体验：游戏库优先、启动入口突出、修复入口可感知。
- 门户体验：详情信息丰富、内容可分享、购买转化路径清晰。
- 无障碍：字号可调、对比度达标、关键流程键盘可达。

## 9. 安全、稳定性与运维
- 安全：权限分级、敏感配置加密、支付回调验签、防刷限流。
- 稳定：关键任务重试队列、下载节点健康探针、失败自动降级。
- 可观测：日志分层、关键指标看板、链路追踪、告警分级。
- 审计：后台配置与支付配置变更必须记录操作者、时间与差异快照。

## 10. 发布与环境策略
- 环境分层：dev / staging / prod，配置与资源地址隔离。
- 发布策略：灰度发布优先，先配置后功能，先小流量后全量。
- 回退策略：版本包与配置可独立回滚，保留最近 N 次发布快照。

## 11. 分期实施建议
- P1：账号、产品展示、下载安装闭环、基础后台配置。
- P2：支付渠道与订单对账、修复中心、客户端启动管理优化。
- P3：灰度热更新、推荐系统、运营活动编排、数据分析看板。

## 12. 成功指标（KPIs）
- 维护效率：非壳层更新中通过后台配置完成的比例 ≥ 80%。
- 交付成功率：下载成功率、安装成功率、修复成功率持续提升。
- 业务转化：详情页到下载、下载到安装、安装到付费的转化增长。
- 稳定性：支付回调成功率、配置发布失败率、严重故障恢复时长达标。
